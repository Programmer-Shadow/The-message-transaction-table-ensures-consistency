## Q1：发货单状态一直是 PENDING，是 bug 吗？

**A**：正常现象，PENDING 是设计终态。

本 Demo 演示的核心是 Outbox Pattern 的可靠消息投递，不是完整的业务状态机。`ShipmentService.createShipment()` 写死 `status=PENDING`，代表"发货单已建、等待拣货"。SHIPPED / DELIVERED 状态推进属于"发货执行"模块，不在 Demo 范围内。

---

## Q2：MySQL UPDATE 都会加行锁吗？

**A**：是的，但有前提——WHERE 条件必须命中索引，否则升级为表锁。

本项目的 CAS 更新：
```sql
UPDATE outbox_message SET status='PROCESSING' WHERE id=#{id} AND status='PENDING'
```
`id` 是主键，命中索引 → 只锁一行。两个调度器实例并发执行时，MySQL 行锁保证串行，`AND status='PENDING'` 保证只有一个能成功（返回 1），另一个返回 0 直接跳过，不会重复发 MQ。

## Q3：JMX Agent 端口冲突，IDE 启动报错

**A**：JMX 是 Java 运行时监控接口（供 JConsole / VisualVM / Arthas 连接），Demo 用不到。

报错原因：IDE 自动加了 `-Dcom.sun.management.jmxremote` 参数，随机分配的端口被其他进程占用。

解决：`Run/Debug Configurations → Modify options → 取消勾选 Allow JMX Agent`。或固定端口：
```
-Dcom.sun.management.jmxremote.port=19999
-Dcom.sun.management.jmxremote.authenticate=false
-Dcom.sun.management.jmxremote.ssl=false
```

---

## Q4：IDE 启动两个服务，第一个会被自动停掉

**A**：IDEA 默认 Single Instance 限制。

解决：`Edit Configurations → 选中配置 → Modify options → 勾选 Allow multiple instances`，两个服务配置都要勾，之后可以同时运行 8080 和 8081。

## Q5：IdempotentService 的 `businessLogic.run()` 跑到哪里去了？

**A**：`businessLogic` 是 `ShipmentConsumer` 传入的 Lambda，`run()` 实际执行 `ShipmentService.createShipment()`。

调用链：
```
ShipmentConsumer.onMessage()
    └── idempotentService.executeIdempotent(messageId, () ->
            shipmentService.createShipment(orderNo, productId, quantity))
                └── businessLogic.run()
                        └── INSERT INTO shipment
```

用 `Runnable` 的好处：`IdempotentService` 不依赖任何具体业务类，任何业务逻辑都能复用同一套幂等保护，实现关注点分离。

---

## Q6：日志颜色、SQL 日志关闭

**A**：两个配置项：

关闭 MyBatis SQL 日志（删除 `log-impl`，改用 logging level）：
```yaml
logging:
  level:
    com.baomidou.mybatisplus: WARN
    org.apache.ibatis: WARN
```

开启 ANSI 彩色输出到文件：
```yaml
spring:
  output:
    ansi:
      enabled: always
```

查看彩色日志：
```bash
less -R +F /tmp/order-service.log
```

## 关键教训

1. **Demo 的边界要清晰**：PENDING 状态不是 bug，是 Demo 范围的终点。看到"异常"先问"这是不是设计意图"。
2. **MySQL 行锁的前提是索引**：CAS 能生效是因为 `WHERE id=?` 命中主键，换成无索引列就退化为表锁，并发性能骤降。
3. **Runnable 作为策略注入是解耦的好方式**：幂等逻辑和业务逻辑完全分离，`IdempotentService` 零业务依赖，可直接复用到其他场景。
4. **IDE 的隐藏配置会干扰调试**：JMX Agent、Single Instance 这类默认开启的选项，不了解时容易误判为代码 bug。
