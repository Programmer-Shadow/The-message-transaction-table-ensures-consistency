# 202604202102-复盘

电商 Outbox 项目在 WSL2 环境下启动联调，从无法发送 MQ 到全链路打通。核心卡点集中在 **WSL2 ↔ Windows RocketMQ 的网络和配置联动**。

---

## Q1：WSL2 里 `docker` 命令找不到

**原因**：重装 Windows 后 WSL 没装 Docker。

**A**：改用 WSL2 原生方式安装：
```bash
curl -fsSL https://get.docker.com -o get-docker.sh
sudo sh get-docker.sh
sudo usermod -aG docker $USER
```

---

## Q2：服务启动报 `Unknown database 'order_db'`

**原因**：Windows 上新装的 MySQL 只有默认库，`order_db` / `warehouse_db` 没建。

**A**：在 JDBC URL 后加 `createDatabaseIfNotExist=true`，Spring 启动时会自动建库：
```yaml
url: jdbc:mysql://localhost:3306/order_db?useSSL=false&serverTimezone=Asia/Shanghai&allowPublicKeyRetrieval=true&createDatabaseIfNotExist=true
```

---

## Q3：订单创建返回 400 Bad Request

**原因**：`CreateOrderRequest` 要求 `userId` 非空，但测试 body 没带。

**A**：正确的请求：
```bash
curl -X POST http://localhost:8080/orders -H "Content-Type: application/json" \
  -d '{"userId":1,"productId":1001,"quantity":1,"orderNo":"xxx"}'
```
另外 `productId` 必须是 `data.sql` 里已初始化的（1001/1002/1003）。

---

## Q4：Outbox 发 RocketMQ 一直失败，`BrokersSent: [LAPTOP-9LRV2QV6]`

**原因**：Windows RocketMQ broker 自动把主机名 `LAPTOP-9LRV2QV6` 注册到 NameServer，WSL2 里这个主机名默认解析到 `127.0.1.1`，而 broker 实际只在 `127.0.0.1` 监听。

**A**：不改 WSL2 hosts，而是改 Windows broker 配置，让它注册明确的 IP：
```properties
# E:\rocketMQ\...\conf\broker.conf
brokerIP1=127.0.0.1
```

---

## Q5：改了 `broker.conf` 没生效

**原因**：`play.cmd` 启动脚本没加载 `broker.conf`，broker 跑的还是默认配置。

**A**：直接在 CMD 里用命令行启动 broker，显式 `-c` 指定配置：
```cmd
mqbroker.cmd -n localhost:9876 -c ../conf/broker.conf autoCreateTopicEnable=true
```
重启后看日志出现 `The broker[broker-a, 127.0.0.1:10911] boot success` 就对了。

---

## Q6：连接通了，但发送还是失败 `CODE: 14`

**原因**：日志里写得很明白——
```
DESC: service not available now, maybe disk full, CL: 0.92 CQ: 0.92 INDEX: 0.92
BROKER: 127.0.0.1:10911
```
不是代码问题，是 RocketMQ 默认存储路径在 `C:\Users\<用户名>\store`，**C 盘使用率 92% > 默认阈值 75%**，broker 主动拒绝写入保护数据。（第一次误判是 E 盘满了，应该先看存储路径）

**A**：把 store 改到有空间的 E 盘：
```properties
# broker.conf
storePathRootDir=E:/rocketMQ/store
storePathCommitLog=E:/rocketMQ/store/commitlog
storePathConsumeQueue=E:/rocketMQ/store/consumequeue
storePathIndex=E:/rocketMQ/store/index
storeCheckpoint=E:/rocketMQ/store/checkpoint
abortFile=E:/rocketMQ/store/abort
```

---

## 关键教训

1. **WSL2 + Windows 服务联调的核心在 IP/主机名解析**。broker 注册的是什么地址，客户端就按这个地址连，主机名侧修复不如服务端直接配 `brokerIP1`。
2. **配置没生效 ≠ 配置错了**。`play.cmd` 是快速启动脚本，不一定读 `broker.conf`，排查时先确认「我改的配置真的被加载了吗」。
3. **RocketMQ `CODE:14` 不一定是 broker 宕机**，`DESC` 里已经写清楚是磁盘水位问题，要看准确的存储路径挂在哪个盘。
4. **错误信息要读完整**。早期我一直盯着 `BrokersSent`，后来才去拿 `Caused by` 才看到真正的根因；下次遇到长栈先拉 `grep -A 30 ... | grep Caused`。
